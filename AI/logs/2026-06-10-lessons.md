## 발생한 문제 및 원인

### 문제 1 — Write 도구의 조용한 파일 절단
**현상:** `Write` 도구로 작성한 파일이 123줄에서 잘림. AST 오류 없이 임포트됨.  
**원인:** `Write` 도구는 약 200줄 이상에서 내용을 조용히 잘라냄.  
**해결:** `bash heredoc` (`cat > file << 'PYEOF' ... PYEOF`)으로 재작성.  
**예방:** 200줄 초과 파일은 항상 heredoc 사용. 작성 후 `wc -l` 확인.

### 문제 2 — stale pyc가 절단된 소스를 숨김
**현상:** `contracts.py`가 659 bytes (실제 절단)인데, pyc(1328 bytes, Jun 9 15:27)가  
완성된 버전을 캐시 → Python이 pyc를 사용해 절단 사실이 드러나지 않음.  
**원인:** pyc 타임스탬프 > 소스 타임스탬프 → Python이 소스 재컴파일 생략.  
**해결:** `touch src/flatten/contracts.py` 로 소스를 pyc보다 최신으로 만들어 재컴파일 강제.  
**예방:** 소스 수정 직후 반드시 `touch`. 임포트 결과가 소스와 다르면 pyc 의심.

### 문제 3 — Read 도구가 pyc 캐시 내용을 "정상"으로 표시
**현상:** `Read` 도구로 `contracts.py`를 읽으면 `call_site` 필드가 있는 것처럼 보임.  
**원인:** Read 도구가 실제 파일 크기(659 bytes)와 무관하게 pyc 기반 메모리 내용을 표시.  
**해결:** `cat`, `wc -c`, `xxd | tail` 로 바이트 레벨 실제 내용 확인.  
**예방:** 의심스러우면 `xxd` 또는 `wc -c` 로 파일 크기를 직접 확인.

### 문제 4 — 테스트 파일도 절단 (pyc에 이전 세션 경로가 기록됨)
**현상:** pytest 출력에 `/sessions/zealous-youthful-ptolemy/...` 경로 표시 → 이전 세션 pyc 사용 중임을 의미.  
**원인:** `test_integration.py`, `test_tracer.py`도 중간에 잘려 있었고 pyc가 구 버전 테스트를 실행.  
**해결:** pyc 바이트코드 역분석 (`marshal.loads` + `dis.dis`) 으로 원본 로직 복원 후 파일 완성.  
**예방:** pytest 출력에 낯선 경로 등장 시 즉시 해당 소스의 `wc -l` + `tail` 확인.
